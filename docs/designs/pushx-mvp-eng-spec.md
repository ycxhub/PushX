### 1.3 State Machine — Workout Session

```
[HOME] ──tapStart──▶ [CAMERA_STARTING]
                          │
                     firstFrame
                          │
                          ▼
                     [TRACKING] ◄──reset──┐
                          │               │
                      tapStop             │
                          │               │
                          ▼               │
                   [SAVING_SESSION]       │
                          │               │
                     save OK / fail       │
                          │               │
                          ▼               │
                   [SUMMARY] ──newSession─┘
                          │
                      tapDone
                          │
                          ▼
                       [HOME]
```

Failure paths:
- Camera permission denied → [HOME] with alert + Settings link
- Camera startup timeout → [HOME] with "retry" prompt
- <2 reps → session saved with nil scores, summary shows "Not enough reps"
- SwiftData save fails → summary still displayed from memory, toast "Save failed"
- App killed during [TRACKING] → data lost (acceptable for MVP)

## 2. SwiftData Models

### 2.1 PushupSession

```
@Model PushupSession
  ├── id: UUID
  ├── startedAt: Date
  ├── endedAt: Date
  ├── repCount: Int
  ├── compositeScore: Int?        (nil if <2 reps)
  ├── depthScore: Int?
  ├── alignmentScore: Int?
  ├── consistencyScore: Int?
  ├── improvements: [String]      (Codable, stored as JSON blob)
  ├── providerType: String        ("MediaPipe" or "Apple Vision")
  └── @Relationship(deleteRule: .cascade)
      reps: [PushupRepRecord]
```

### 2.2 PushupRepRecord

```
@Model PushupRepRecord
  ├── repNumber: Int              (1-indexed)
  ├── durationSeconds: Double
  ├── depthScreenSpace: Double    (minNoseY − maxNoseY in normalized coords)
  ├── depthWorldMeters: Double?   (maxWorldY − minWorldY, meters, nil if Apple Vision)
  ├── shoulderAsymmetry: Double   (avg |leftShoulderY − rightShoulderY| across paired samples)
  ├── shoulderAsymmetryWorld: Double?  (same, world coords, nil if Apple Vision)
  └── session: PushupSession?     (inverse relationship)
```

### 2.3 RepMeasurement → PushupRepRecord Mapping

The existing `RepCountingEngine.RepMeasurement` stores per-frame arrays:
- `leftShoulderYs: [CGFloat]`, `rightShoulderYs: [CGFloat]` (screen)
- `leftShoulderWorldYs: [Float]`, `rightShoulderWorldYs: [Float]` (world)

These can be hundreds of values per rep. The mapping computes summary stats:

```
shoulderAsymmetry = mean(|leftShoulderYs[i] − rightShoulderYs[i]|) for i in 0..<min(left.count, right.count)
depthScreenSpace = minNoseY − maxNoseY
depthWorldMeters = (maxWorldY − minWorldY) if both non-nil, else nil
```

This reduces per-rep storage from ~400 floats to 5-6 doubles.

### 2.4 Minimum iOS Version

**SwiftData requires iOS 17.0.** The PRD targets iOS 16+, but SwiftData forces a bump to 17.0. This is acceptable:
- iOS 17 released Sept 2023 — 2.5 years old by ship date
- Minimum device: iPhone XS (2018) — 8 years old
- The WhatsApp group of 100 members will overwhelmingly be on iOS 17+
- Swift Charts (for trend view) requires iOS 16+ — also covered

**Decision: Bump deployment target to iOS 17.0.**

```json
    "duration_seconds": 45.2,
    "rep_count": 12,

    "scores": {
      "composite": 78,
      "depth": 82,
      "alignment": 71,
      "consistency": 80
    },
    "improvements": [
      "Keep your shoulders level.",
      "Try going deeper on each rep."
    ]
  },
  "reps": [
    {
      "number": 1,
      "duration_s": 2.1,
      "depth_normalized": 0.15,
      "depth_meters": 0.12,
      "shoulder_asymmetry": 0.02,
```


## 4. Error & Rescue Map

```
METHOD                          | FAILURE MODE              | RESCUED? | ACTION                              | USER SEES
--------------------------------|---------------------------|----------|-------------------------------------|----------------------------------
SessionStore.save()             | SwiftData write error      | Y        | Log error, keep scores in memory    | Summary shows; toast "Save failed"
SessionStore.fetchAll()         | SwiftData read error       | Y        | Return empty array                  | Empty history, no crash
FormScoring.computeScores()     | <2 reps (returns nil)      | Y        | Session saved with nil scores       | "Not enough reps for scoring"
SessionExporter.toJSON()        | Encoding error             | Y        | Return error string                 | "Export failed" alert
Camera permission               | Denied by user             | Y        | Show alert with Settings deep link  | "Camera access required" + button
Camera startup                  | AVCaptureSession timeout   | Y        | Show retry button                   | "Camera didn't start" alert
Camera startup                  | No front camera (rare)     | Y        | Show error                          | "No front camera found"
ModelContainer init             | Schema migration failure   | Y ← GAP  | Need: catch and reset store         | "Data reset" alert ← MUST HANDLE
SwiftData @Query                | Background thread access   | N/A      | @Query runs on @MainActor           | N/A
```

**CRITICAL GAP:** `ModelContainer` initialization can fail if the SwiftData schema changes between app versions. For MVP this won't happen (first install), but add a `do/catch` around container creation with a fallback that deletes and recreates the store. Flag for future: proper schema versioning.

## 7. Performance Review

| Concern | Analysis | Action |
|---------|----------|--------|
| RepMeasurement memory | Per-frame shoulder arrays: ~100 floats/rep × 50 reps = 5000 floats = ~40KB | Acceptable for session lifetime. Summarized to 5 doubles/rep on save. |
| SwiftData write on session end | Single write: 1 session + ~20 rep records | <50ms. Non-blocking if dispatched after UI update. |
| SwiftData query for history | MAX ~200 sessions in first 3 months (3x/day × 90 days × 2 sets = 540) | Simple fetch with sort descriptor. No index needed at this scale. |
| Swift Charts rendering | MAX ~200 data points for trend chart | Trivial for Swift Charts. |
| N+1 queries | HistoryView lists sessions (no rep details). SessionDetailView fetches one session's reps. | No N+1 risk — reps loaded lazily via relationship. |
| Camera frame processing | Unchanged from Phase 0 — no new work per frame | No regression. |
